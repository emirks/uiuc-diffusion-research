The ontology

Your key idea: an endpoint carries no transition — it's just content. So novelty lives on the transition source (the reference / specialty weights), and content is a separate axis. Endpoints are always untrained content (test-band / held-out / DAVIS), sidedness-matched — except the two fit anchors.

Reference novelty (training exposure of the transition source)
  • seen — the exact demo was in training (specialist weights, or a generalist reference = a train-band clip of a held-in class)
  • unseen — class trained, but this demo instance never was (a test-band clip of a held-in class) → new demo of a known transition
  • zero-shot — a reference from a held-out class (never trained)
Specialists have only seen/unseen on endpoints — they know one transition, so no reference-novelty axis.
Content relationship (endpoint vs transition class)
  • same — endpoint from the transition's own class
  • cross — endpoint from a different corpus class (sidedness-matched)
  • foreign — endpoint from DAVIS (off-distribution real footage)
Cross pools held-in-test + held-out endpoints but tags endpoint_source — the held-in-vs-held-out contrast (interference) is recoverable.

G-ref-control is new since the proposal and it is load-bearing: without it, G-unseen-same and G-memo-probe cannot distinguish "the model used the in-context demo" from "the endpoint and the token were enough." It is byte-identical to its G-unseen-same twin except the demo, which comes from a different class, and it is scored against the same pool — so the pair isolates the demo's contribution.
The two generalization gaps this buys: seen→unseen = G-memo-probe − G-unseen-same (endpoint held fixed) = demo-instance memorization; unseen→zero-shot = class generalization. That's the capability-vs-lookup distinction. Free headline the grid also enables: SP-same vs G-unseen-same on the same class = "does in-context match the specialist's baked-in ceiling?" — the cost of IC vs specialization (derived, no new gens).

Rules (asserted at build)

  • Novelty = f(ref) only. An endpoint's train/test/held-out split affects content-novelty, never the seen/unseen/zero-shot label. A held-out clip used as a cross endpoint under a held-in donor is unseen-cross — its held-out-ness doesn't leak.
  • Strict sidedness-matching (one↔one, two↔two): endpoint sidedness == transition-source sidedness. Never a two-sided transition between one-sided endpoints (a one-sided class's end frames are its effect — verified).
  • Content class-disjointness: same ⟹ endpoint_class==donor & clip≠ref; cross/foreign ⟹ endpoint_class≠donor (guards zero-shot self-reference).
  • Only the two fit anchors (SP-fit, G-fit) carry train endpoints; every other cell draws from {test, held-out, DAVIS}.
  • Holdout FINALIZED (split v1.2): 10 held-out classes, none specialists — transform: firelava, melt_transition, raven_transition, luminous_gaze · overlay: saint_glow, monstrosity, cotton_cloud · cover: display_transition · traverse: flying_cam_transition, live_concert. So zero-shot n=10 (bar 9/10). gas/hero/illustration returned to training; jump/seamless dropped (1 clip).
  • G-unseen-same n=13 (held-in classes with ≥2 test clips, up from 8) after the split-v1.2 test-clip rule promoted earth_element_6 + money_rain_1 to test.

Non-leaking prompts

Builder-rendered from the conditioning; a neutral token holds the transition slot. One-sided "{S1}. sksz." · two-sided "{S1}. sksz. {S2}." (trailing punctuation stripped; unit-tested). DAVIS endpoints get ~10–15 hand-written static captions, run through the same audit.

Token choice. Default sksz (2 Gemma sub-words, ~zero prior, 0 corpus collision), probe-gated. TR4NS1T10N was tested and rejected: Gemma-3 splits it into 8 subwords (TR·4·NS·1·T·1·0·N) — far past the ≤2 criterion — and it re-evokes "transition," which would let the base model do transition work from the token and muddy the text_floor leak-proof. Neutral short tokens keep that proof clean.

Registry & structure

Sources of truth: corpus manifest + splits.json + arms.yaml + builder. registry.jsonl = atomic build output; cell labels derived from stored ref_split, endpoint_split, held_in, donor_class, endpoint_source, sided. base_key = endpoint+prompt+sided+seed (base has no novelty → dedup by input-hash). Every % row carries a %_type ∈ {same, proxy}.

Evaluation & metrics

Claim channels
appearance-margin + 2AFC.
GT pool
the class the arm should produce = reference/specialty class (donor for cross/foreign).
% typing
%_same (endpoint class == donor): fair, cross-class comparable, headline-eligible. %_proxy (cross/foreign/DAVIS): content-capped donor-proxy — within-cell ranking/monitoring only, never a headline magnitude, cross-cell comparison, or aggregate ladder-%. Hard-asserted, not a footnote.
Cross/foreign headline
margin Δ = treat − base on the same (content, ref, sidedness) input + 2AFC. Base eats the same content-cap, so it cancels — a low DAVIS % is harmless to the claim.
Dominance (cross)
score vs donor pool AND recipient-native pool → 2×2: reference won / endpoint-prior won / mush.
Mask
pure function of the row's conditioning dict, not the class label.
Copy-guard
pool excludes the reference clip + endpoint clip (verified).
Seeds
≥2 per eval item.
Your metric question: app_ref is pairwise (gen vs one reference). The pool score = mean over the class pool = closeness to the class as a whole (mean-of-pairwise, not centroid). The % = pool / class-ceiling; the ceiling carries the same class-spread penalty so it cancels — only %_same is cross-class comparable, and >100% is legal. Stats on raw; % is presentation.

The report table — % everywhere, claims still typed

You wanted the percentage to be the headline. It is — with one rule that keeps it honest. run_eval.py --mode report emits a single table and hard-refuses to aggregate across % types:

cell              n  %type   level %   Δpp vs base   donors +   verdict
SP-same          22   same     87.4%         +21.3      10/11     positive
SP-cross         22  proxy   (41.2%)         +18.7       9/11     positive
G-unseen-same    13   same     81.0%         +15.9      11/13     positive
G-ref-control    13   same    (…)            …           …        control
                                    ^ greyed: content-capped level, ranking only
  • Same-content cells (SP-same, G-memo-probe, G-unseen-same, G-zs-same): the absolute %_same IS the headline, directly comparable across classes.
  • Cross / foreign cells: the headline is Δ percentage-points = %proxy(treat) − %proxy(base) on the same input. Both arms are capped identically by the borrowed content, so the cap subtracts out and the Δpp is a fair, %-denominated effect size. The absolute level is still shown — greyed, labelled ranking-only.
  • Unit of analysis is the class: per-donor mean of the margin, then a one-sided sign test.

Numbers in the sample table above are illustrative formatting, not results — nothing is scored yet.

Queue policy. secondary is the workhorse (preemptible, 4 h cap): every ladder2 job is resumable — training resumes from checkpoints every 250 steps, generation skips existing outputs — so a preemption costs minutes. Short walltimes are requested deliberately because they backfill sooner. The lab HCESC partitions are shared with teammates: never more than 4 ladder2 jobs there at once.

Decisions taken (and the ones you should know about)

  1. The fleet is gated ~2–3 h behind a 2-model pilot. Everything else parallelises; the leak-free training prompt is the one genuinely new variable, and a systematic defect would otherwise cost a 12-model rerun. This is a deliberate, small deviation from "everything at once".
  2. Cross / foreign headlines are Δ percentage-points vs base, not absolute %. The compliant version of "base it on the percentage": the identical content cap cancels in the difference. %_same stays the absolute headline for same-content cells.
  3. Lab-partition plan: the generalist (~5 h, does not fit secondary's 4 h cap) + a short probe + at most 2 rescue slots — never more than 4 at once. Everything else on secondary with requeue + resume.
  4. Abort rule, pre-committed: if a pilot specialist shows no in-distribution effect onset by step 1000, the fleet stops and the filmstrips come back for judgment — reverting to leaky prompts is not an available fix.
  5. Training prompt == inference prompt, enforced structurally: the same render_prompt() call emits the training captions and the registry rows. Every prior conditions/ tree is discarded (they hold the leaky caption).
